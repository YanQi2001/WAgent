# WAgent 简历修改建议

## 一、当前简历中 WAgent 部分的原文

```
Agent 项目                                         2025.11 - 2026.2
基于 LangGraph 的 AI 面试智能体系统
1. Harness-Centric 架构：Harness 外层控制 + LangGraph 内层推理
2. Raw/Compacted/Summarized 三级上下文压缩 + Progress File 注入，节省 token
3. RAG：BM25 + Dense + RRF + Cross-Encoder；Anthropic Contextual Retrieval
4. GAP 分析 + 动态 taxonomy
5. 多 Agent：Router/Interviewer/Judge/Knowledge Updater/Intent Classifier
```

## 二、存在的问题

1. **缺少项目链接**：没有 GitHub 仓库地址，面试官无法查看代码
2. **缺乏量化指标**：全是技术名词堆砌，没有数字衡量项目规模和效果
3. **技术细节堆砌但亮点不突出**：5 条都是"用了什么技术"，但"解决了什么问题、效果如何"几乎没有
4. **遗漏了大量实际实现的特性**：双模型分级调用、小红书 MCP + OCR 管线、CJK 终端适配、守护进程自动更新等都没提及
5. **项目名称和定位偏弱**："Agent 项目"太泛，没有体现工程完整度

## 三、可用的量化指标

基于项目当前实际状态统计：

| 指标 | 数值 | 说明 |
|------|------|------|
| 核心代码量 | ~5,000 行（有效 Python） | `src/wagent/` 去空行去注释 |
| 项目总代码量 | ~14,000 行 | 含 tools、tests |
| 智能体数量 | 5 个 | Router / Interviewer / Judge / Intent Classifier / Knowledge Updater |
| RAG 检索管线层数 | 4 层 | BM25 → Dense → RRF 融合 → Cross-Encoder 精排 |
| MCP 数据源 | 3 个 | 小红书（含图片 OCR）、Bing、PDF |
| 知识库 Topic 覆盖 | 34 个动态 topic | 由 LLM 自动扩展的 taxonomy |
| 自动入库 QA 对 | 800+ 条 | 来自小红书/Bing 联网搜索，经 LLM 质量过滤 |
| Topic 覆盖领域 | 28 个 | 实际有入库内容的 topic 数 |
| 核心依赖 | 18 个 | LangChain / LangGraph / Qdrant / RapidOCR 等 |
| CLI 命令 | 10+ 个 | start / interview / qa / ingest / serve / topics / purge 等 |

## 四、建议修改版本

```
WAgent — AI 面试智能体系统（开源）              2025.11 - 2026.3
GitHub: github.com/YanQi2001/WAgent
独立设计并实现的工业级 AI 面试模拟系统，核心代码 5,000+ 行，
覆盖多智能体协作、RAG 知识库、MCP 外部数据源和自动化知识更新全链路。

1. 架构设计：提出 Harness-Centric 架构，外层 Harness 管理中间件管道、
   Token 预算和三级上下文压缩（Raw → Compacted → Summarized），
   内层委托 LangGraph 状态机驱动 5 个智能体协作；简历驱动模式注入
   候选人简历原文进行项目深挖式追问（技术选型 / 实现细节 / 困难与
   解决），随机八股模式覆盖基础知识广度

2. 混合检索 RAG：实现 4 层检索管线（BM25 稀疏 + bge-large-zh 稠密 →
   RRF 融合 → bge-reranker-v2-m3 精排），引入 Contextual Retrieval
   为每个 chunk 生成 LLM 上下文描述后再向量化，构建覆盖 34 个动态
   topic 的结构化知识库

3. MCP 数据源 + OCR：对接小红书、Bing 等 3 个 MCP 数据源，集成
   RapidOCR（ONNX/CPU）提取帖子图片文字，实现 GAP 分析 → 联网搜索
   → LLM 质量过滤 → 语义去重的自动更新管线，已累计入库 800+ 条
   高质量 QA 对

4. 双模型分级调用：设计 Fast/Strong 两级 LLM 调度，简单任务
   （意图分类、元数据生成等 8 个调用点）走廉价模型，复杂任务
   （QA 提取、面试对话等 7 个调用点）走强推理模型，降低 API 成本
```

## 五、修改要点说明

### 5.1 结构优化

- **第一行**：项目名 + 定位 + "（开源）"标签 + 时间
- **第二行**：GitHub 链接，方便面试官直接查看
- **第三行**：一句话概述 + 核心量化数字（5,000+ 行），建立规模感

### 5.2 四条描述的策略

每条遵循 **"做了什么 → 怎么做的 → 效果/规模"** 三段式：

| 条目 | 核心亮点 | 量化 |
|------|---------|------|
| 架构设计 | Harness-Centric 原创架构 + 三级压缩 | 5 个智能体 |
| 混合检索 RAG | 4 层管线 + Contextual Retrieval | 34 个动态 topic |
| MCP + OCR | 全链路自动更新 + 图片文字提取 | 3 个数据源、800+ QA 对 |
| 双模型分级 | Fast/Strong 调度降成本 | 8 + 7 = 15 个调用点分级 |

### 5.3 删掉了什么

- "GAP 分析 + 动态 taxonomy" 作为独立一条太薄，合并到第 3 条的自动更新管线中
- "Progress File 注入" 过于细节，简历中不需要

### 5.4 新增了什么

- GitHub 链接
- 代码规模量化（5,000+ 行）
- MCP + OCR 全链路（原版完全没提）
- 双模型分级调用（原版完全没提）
- 入库数据量化（800+ QA 对、34 个 topic）
